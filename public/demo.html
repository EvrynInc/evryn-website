Ephemeral Session
No cookie-based session persistence. Every new page load = fresh conversation. Session lives in a server-side session store, keyed to a token in the tab's JavaScript runtime — dies when the tab closes. If user closes the tab and returns, Evryn doesn't remember them. This is a deliberate safety choice: prevents Person A's data from leaking to Person B on a shared device. User already consented to ToS/PP at the landing page (Screen 1).
Data Processing
All conversation data processed through Anthropic Claude API. If user never creates an account, conversation content is discarded. IP/behavioral data may be retained separately for security (DDoS, repeated attacks) — security infrastructure, not user data. Whether any conversation content retention is needed for the security window is an open question.
User Isolation
User isolation is absolute. Evryn never reveals one user's information to another — including whether she knows someone, has spoken with them, or has any relationship with them. Even if a user mentions another person by name, Evryn treats each conversation as fully independent. Evryn does not confirm or deny relationships with other users.
User Recognition
Evryn may recognize a user from publicly available information only. Here, she references NWFF's coverage of Jamie's film — public info. Evryn will only volunteer recognition when the match is unambiguous (specific name + specific work, clearly public). If there's any uncertainty, she doesn't volunteer — she lets the user tell her who they are.

Consent — Second Gate (Belt-and-Suspenders)
Second consent event at account creation, with path-appropriate UX. OAuth: tapping Google/Apple shows inline consent confirmation — user agrees before redirect. Email: checkbox gates the Create account button. Both paths produce a logged consent event. No account can be created through any method without affirmative consent. Even if someone bypassed the pre-chat checkbox (e.g., direct link to chat), every account holder has agreed to terms with 100% certainty. Consent logged: timestamp, auth method (Google/Apple/email), document versions, user ID.
Conversation Migration
If account is created within the same session, the entire pre-account conversation migrates into the new user profile. Understanding Evryn built during ephemeral session becomes the foundation of the permanent profile. User picks up right where they left off — account creation is a brief interruption, not a restart.
Data Collection
Auth credentials collected (email or OAuth token). Stored via Supabase Auth (hosted PostgreSQL, encryption at rest and in transit).

Consent — Progressive
Neither party learns the other exists unless both opt in. Some connections need one round; others need several with progressively more detail (e.g., first name, photos in dating contexts). Evryn reads the context and adjusts. Each round is a separate consent event — user reviews exactly what Evryn would share about them and approves before anything is sent. Users can withdraw at any point — match becomes a non-event. Full name and contact info are not shared before payment. Consent logged: timestamp, user ID, description version, what information was included.
Matching — AI Decision
Match determined by structured filters (geography, profession, constraints) and profile similarity analysis, with final evaluation by Evryn. Match descriptions written by Evryn — her voice, her assessment, her framing.

Consent — Identity Verification
Separate consent event, distinct from ToS/PP consent. Consent logged: timestamp, user ID, verification provider.
Third Party — iDenfy
Identity verification handled by iDenfy (document + selfie match). Evryn stores ONLY: verified/not-verified flag, verification timestamp, and a non-reversible trust fingerprint hash. No ID documents, selfie images, or document numbers stored by Evryn. iDenfy's own data retention policy governs document storage on their end.
Trust Fingerprint Hash
On successful verification, a deterministic non-reversible hash is generated from the verification result data (not the raw documents, which are never stored by Evryn). The same person re-verifying will produce the same hash. Persists after account deletion — prevents bad actors from deleting and re-creating accounts to reset trust history.

Payment processed via Stripe. Simple pay-in-the-moment — no Evryn Wallet, no pre-purchased credits, no installment plans in v0.3. Evryn does not store payment card details — Stripe handles PCI compliance. No sensitive transaction info is passed to the Anthropic API.
